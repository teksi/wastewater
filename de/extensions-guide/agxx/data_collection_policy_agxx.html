5.2.2. Erfassungsrichtlinie

5.2.2.1. Mapping

Die AG-64/AG-96 Object-ids werden wie folgt in die Datenbank gemappt:

  • Infrastrukturknoten/GEPKnoten wird auf tww_od.wastewater_node gemappt

  • Infrastrukturhaltung/GEPHaltung wird auf tww_od.reach gemappt

  • Ueberlauf_Foerderaggregat wird auf tww_od.overflow gemappt

  • Einzugsgebiet wird auf tww_od.catchment_area gemappt

  • SBW_Einzugsgebiet wird auf tww_od.catchment_area_totals gemappt

  • GEPMassnahme wird auf tww_od.measure gemappt

  • VersickerungsbereichAG wird auf tww_od.infiltration_zone gemappt

Wo möglich werden die Wertelisten von VSA DSS übernommen In den Fällen, in denen ein 1:1 mapping möglich ist, existiert kein zusätzlicher Wertelisteneintrag für AG-64/96. Stattdessen kann der entsprechende VSA-DSS-Wert verwendet werden, welcher beim Export übersetzt wird. Die entsprechenden Fällis sind unten aufgelistet

Almost all VSA-DSS values of the mentioned layers that do not exist in AG-64/96 are mapped to AG-64/96 using a backwards relation. Per default, the VSA-specific value list codes which are ambiguous in AG-64/96 are set to inactive.

AG-64/96-Werte werden wo modelltechnisch sinnvoll bei einem Export automatisch auf VSA DSS gemappt, was es erlaubt, beide Modelle zu exportieren.

5.2.2.1.1. Umgang mit Organisationen

Die Organisationstabelle von AG-64/96 unterscheidet sich von derjenigen des VSA. Sie hat 20 Zeichen und nutzt einen anderen Präfix. Um die VSA-Komptibilität zu wahren nutzt TWW 2 AG-64/96 die Tabelle des VSA.

Einige private Organisationen in der AG-64/AG-96 Tabelle wurden nicht in die Tabelle des VSA übernommen. Für diese Organisationen wird ein separater OID Präfix, welcher spezifisch für diesen Zweck generiert wurde, und der AG-64/AG-96 Postfix. genutzt Die entsprechenden Daten werden bei der Initialisierung eingelesen.

Beim Export eines Datensatzes werden die Organisationen nich aus der Datenbank exportiert. Stattdessen ist die AG-64/96-Organisationstabelle im Plugin hinterlegt und wird zum Export-Datensatz hinzugeladen.

5.2.2.1.2. Letzte_Aenderung

In AG-64/96 werden die Daten der Letzten_Aenderung der Werkinformation (WI) und des GEP separat geführt. Diese Werte werden datenbankseitig verwaltet. In den Einstellungen gibt es `hier<./plugin_setup_agxx.html>`_ ein Steuerelement, welches genutzt werden kann, um pro Nutzer zu steuern, welche Werte aktualisiert werden sollen.

  • special_structure

  • discharge_point.relevance

5.2.2.2.1. Umgang mit Hausanschlüssen

Hausanschlüsse sind im Datenmodell als Infrastrukturknoten/GEPKnoten mit FunktionAG „Anschluss“ definiert. Da sie aber keine Abwasserbauwerke sind, ist die Funktion auf dem Abwasserknoten zu attributieren (wastewater_node.ag64_function).

5.2.2.2.2. Umgang mit Deckeln

Die AG-64/AG-96 Erfassungsrichtlinien schreiben folgendes vor:

  • The main wastewater node of a manhole or special structure must be mapped to the location of the main cover.

  • Zweitdecket sind als Infrastrukturknoten/GEPKnoten mit Funktionag „andere“ gemappt.

As topological relations to a node of funktionag „andere“ are technically possible, we need to link all covers to a wastewater node using tww_od.cover.ag64_fk_wastewater_node. In order to follow these limitations, there is an additional foreign key on tww_od.cover pointing to tww_od.wastewater_node. A wastewater node’s situation geometry is only overruled if it is referenced from a cover. Additionally, the attribute wastewater_node.ag64_function can be set to „andere“.

5.2.2.2.3. Handling of the attribute „IstSchnittstelle“

AG-96 has an attribute IstSchnittstelle which is not present in VSA DSS 2020.1. The value is stored in tww_od.wastewater_node.ag96_isgateway.

5.2.2.3. Infrastrukturhaltung/GEPHaltung

Apart from street water and square water, the NutzungsartAG are not modelled as a value list extensions. Use the backwards relation instead.

5.2.2.4. Ueberlauf_Foerderaggregat

The layer Ueberlauf_Foerderaggregat is mapped to tww_od.overflow and its specialisations

5.2.2.6. Bautenausserhalbbaugebiet

Bautenausserhalbbaugebiet is mapped to tww_od.building_group. There is no backwards mapping from VSA-DSS Gebaeudegruppe.Sanierungsbedarf to AG-96 Bautenausserhalbbaugebiet.Sanierungsbedarf because the value unbekannt cannot be mapped.

5.2.2.7. SBW_Einzugsgebiet

SBW_Einzugsgebiet is mapped to tww_od.catchment_area_totals. The perimeter geometry is stored as an extension geometry attribute (ag_96_perimeter_geometry). In order to alter it, one needs to manually import the layer into the qgs project.

There exists a function to calculate the perimeter geometry by aggregating the catchment areas via catchment_area->log_card->main_log_card->hydraulic_char_data->catchment_area_totals. The perimeter geometry is a MultiSurface, while the INTERLIS model requires a CurvePolygon. According to the official data collection policy of the Canton, one should violate the datamodel and export a MultiPart. As the underlying export mechanism ili2pg does not allow to export a wrong geometry type, only the biggest Singlepart is exported.

5.2.2.8. VersickerungsbereichAG

VersickerungsbereichAG is mapped to tww_od.infiltration_zone.
